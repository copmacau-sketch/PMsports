------------------------------------------------------------------------------

2. Each game `<ec>` was missing `<lid>` (league id). The worker reads

   pushes `undefined` into the league array.

3. `now_model` was hard-coded to "FT". The pre-game template
   (`tpl/member/ft_early_game.html`) only ships `model_HT`, not `model_FT`,
   so `getRatioLayer()` returned an empty layer and every game was skipped
   via the `if (!tmpLayer) continue;` guard.

4. `retimeset` used raw minute strings like "10". The worker's `transRETIME`
   crashed on `tmpHash[1].replace(...)` when the format wasn't
   `<Start|LIVE|MTIME|HT>` or `<half>^<minute>'`.

5. Datetime used "MM/DD HH:mm" with slashes. `transDate` splits on "-" and
   produced "05وœˆundefinedو—¥".

6. Odds field names didn't match what the worker reads:
   - `*RATIO_RH*` / `*RATIO_RC*` strip the trailing H/C â†’ worker reads
     `ratio_r` (single field), not `ratio_rh` / `ratio_rc`.
   - The `+`/`-` sign comes from `tmp_game["strong"]` (single field with
     value "H", "C" or ""), not from `strong_reh` / `strong_rec`.
   - 1X2 uses `ior_mh` / `ior_mc` / `ior_mn` (M = money line), not
     `ior_rh` / `ior_rc` / `ior_rd`.
   - Over/Under uses `ior_ouh` (under) / `ior_ouc` (over) with line
     value in `ratio_ouo` / `ratio_ouu`.

2026-05-30 â€” Live odds cache stuck at hours-old (cron mk 3 vs xmlUpd 40)
==============================================================================

SYMPTOM
-------
Active in-play matches were returning phase=closed from activeOddsContext()
in the H5 detail page even though FastAPI status showed status=inplay.  Of
~900 cached files in /dev/shm/crown_limitless_mk/, only 3% were <240s fresh
(cron TTL).  The 2-min ingest_limitless.php cron logged a consistent
"events=60 activeMarkets=13 ... mk 3 ; skipped 19" â€” only 3 cache files
written per run despite 40 r_cn xml UPDATEs.

ROOT CAUSE
----------
File-ownership war between two writers of the same directory:

  - PHP cron (ingest_limitless.php) runs as **www-data**, writes mode 0644.
  - FastAPI WS bridge (oddsapi_ws.py / pmppm-com-api.service) runs as **root**,
    writes mode 0644.

When the cron later tries to overwrite a file already owned by root with
mode 0644, file_put_contents returns false silently (only the file owner
can rewrite a 0644 file).  Over time the WS bridge accumulates root-owned
files in the cache dir; the cron's effective coverage shrinks until only
brand-new gids (created by the cron itself) get refreshed.  Snapshot:

    Before remediation: 536 www-data + 367 root + 1 ?
    After  chown -R:    902 www-data +   1 root

  Cron output flipped immediately:
    09:24:04 ... mk 3   skipped 19    (last hung run, 367 files unwriteable)
    09:26:04 ... mk 37  skipped 19    (after one-shot chown â€” full success)

FIX
---
1. crown-explorer/backend/app/oddsapi_ws.py
   - Added _resolve_www_data_ids() + _chown_to_www_data() helpers.
   - _write_limitless_mk() now chowns its output to www-data:www-data and
     chmod 0664 after every successful tmp.replace(target).  This makes
     subsequent cron writes succeed even though FastAPI still runs as root.
   - Falls back to chmod 0666 if www-data uid lookup fails (still allows
     cron to overwrite).  /dev/shm is local-only so the broader perm is
     acceptable.

2. One-shot prod remediation:
       chown www-data:www-data /dev/shm/crown_limitless_mk/*.json
       chmod 664              /dev/shm/crown_limitless_mk/*.json
   No service downtime; existing root-owned files immediately became
   overwriteable by the next cron tick.

VERIFICATION
------------
- Cron mk count jumped from 3 â†’ 37 on the very next run after chown.
- Cache freshness <240s rose from 23/866 (3%) toward the expected
  ~37 fresh files per cron run for events the upstream actively covers.
- WS bridge still writes via _write_limitless_mk â†’ permissions now stick
  because chown happens on every write.

KNOWN UPSTREAM LIMITATION (not addressed)
----------------------------------------
limitless.show /api/quote returns ~60 events with ~13 having active
markets at any moment.  foot_match has ~67 in-play matches; 30+ of those
will never get fresh limitless odds because the upstream simply doesn't
cover them.  PHP correctly returns phase=closed for those â€” the safety

The /ws/quote firehose itself runs at ~0.23 frames/sec (1 frame every

2026-05-30 â€” Authoritative live ticker (api-sports.io elapsed/status plumb)
==============================================================================

PROBLEM
-------
The H5 live ticker (`formatLiveTicker` in sports/page.tsx) used a
wall-clock heuristic on `commenceTs` to derive game minute and HT/2H
state.  This meant:
  * 45+' / "ن¸­هœ؛" / 90+' transitions fired on fixed wall-clock thresholds
    (50 min, 67 min, 112 min) regardless of actual game state.
  * A real "HT" reached at minute 47 still showed "45+'" until 50 min.
  * Long stoppage time stretched the display past full-time before the
    real broadcast did.
  * api-sports.io ALREADY provides `fixture.status.elapsed` (authoritative
    minute) and `.short` (period code: 1H/HT/2H/ET/FT/AET/PEN/...) but it
    was being read in the cron only for is_live/isFinished decisions and
    then thrown away.

FIX
---
Plumb api-sports' authoritative game state through the entire stack:

1. **Schema** (db_sports.foot_match):
       ALTER TABLE foot_match
         ADD COLUMN apisports_elapsed SMALLINT NULL AFTER inball_c_hr,
         ADD COLUMN apisports_status  VARCHAR(8) NULL AFTER apisports_elapsed,
         ADD COLUMN apisports_seen_at INT NULL AFTER apisports_status;

2. **Cron writer** (`wwwroot_F5PEa/application/bdata/ingest_odds_api.php`):
   - Read `$fix['fixture']['status']['elapsed']` alongside the existing
     `['short']` extraction.
   - Add `apisports_elapsed`, `apisports_status`, `apisports_seen_at` to
     the upsert fields list (works in both full and `--live-only` mode;
     the live-only field-strip list does NOT include the new columns).

3. **FastAPI** (`crown-explorer/backend/app/routers/external.py`):
   - `_MYSQL_EVENT_COLS` SELECT includes the 3 new columns.
   - `_mysql_row_to_event()` surfaces them as `elapsed_minute`,
     `status_short`, `apisports_seen_at` in the JSON event dict.
